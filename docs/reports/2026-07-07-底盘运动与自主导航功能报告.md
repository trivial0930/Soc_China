**范围**:本报告覆盖底盘运动控制、里程计、整车安全、遥控、建图、定位与自主导航等全部功能模块及其关键技术攻关。

## 1. 概述

本机器人为室内巡检机器人的**运动与导航子系统**,以 RDK X5 作为上位机(运行 ROS 2 Humble),STM32F411 作为下位机实时运动控制器,四轮麦克纳姆全向底盘,配备 360° 激光雷达与九轴 IMU。系统实现了从**底层电机闭环控制**到**上层自主导航**的完整能力链:遥控建图 → 地图存储 → AMCL 定位 → Nav2 全局/局部规划 → 动态避障 → 自主到达目标点。整套软件采用一键 launch 启动,配合看门狗、雷达安全门控、断链自愈等多重安全机制,可在真实实验室环境稳定运行。

## 2. 系统架构

### 2.1 硬件构成

- **上位机 RDK X5**:运行 ROS 2 Humble,负责建图、定位、路径规划、避障决策、遥控接收。
- **下位机 STM32F411CEU6**:50 Hz 实时运动控制,四轮速度闭环、里程计采集、安全看门狗。
- **电机驱动**:2× TB6612 双路驱动,驱动四个带霍尔编码器的直流减速电机。
- **底盘**:四轮麦克纳姆,支持前后、左右横移、原地旋转的全向运动。
- **传感器**:镭神 N10 360° 激光雷达(建图/定位/避障)、BMI088 六轴 IMU(偏航角速度融合)。
- **供电**:12 V 动力电池 → 稳压模块 → 5 V 逻辑电,STM32/RDK 逻辑电与电机动力电分离。

### 2.2 软件链路(TF 树)

```
建图:  map ─(slam_toolbox)→ odom ─(EKF)→ base_link ─(URDF静态)→ laser/imu/wheels
导航:  map ─(AMCL)────────→ odom ─(EKF)→ base_link ─(URDF静态)→ laser
数据流: /scan(雷达) + /odom(轮式) + /imu(陀螺) → EKF → 定位/建图 → /cmd_vel → bridge → STM32 → 电机
```

上位机与下位机通过 **USB CDC(Type-C 原生 USB)** 高速链路通信,自定义二进制协议帧(SET_MODE / HEARTBEAT / CMD_VEL / STATUS / ODOM / SET_PID),CRC 校验,`stm32_bridge` 节点负责 ROS ↔ 串口双向翻译。

## 3. STM32 固件功能

### 3.1 麦轮运动学与速度闭环 PID

- **逆运动学 mix**:将机体速度指令 (vx, vy, wz) 解算为四轮角速度,支持全向运动。
- **四轮速度闭环 PID**:50 Hz 控制环,每轮独立 PID(Kp=15 / Ki=30 / 前馈 ff = pwm_max/max_radps),前馈项复现开环特性、PID 消除稳态误差。
- **测速处理**:硬件定时器正交编码器模式(TIM1/2/4/5,4 倍频),编码器增量 → 轮速,配 EMA 低通滤波抑制量化噪声;中速限幅 0.4 m/s 匹配上位机速度上限。
- **运行时整定**:`SET_PID` 协议帧支持运行时在线调参,配 `pid_tune.py` 脚本,免烧录即可整定。

### 3.2 USB CDC 高速链路

底盘链路由 40-pin UART 迁移至 **Type-C 原生 USB CDC**(921600),解决了引脚硬冲突导致的反复失联问题,通信更稳更快。

### 3.3 独立看门狗(IWDG)—— runaway 根治

寄存器级独立看门狗(LSI /32,~250 ms 超时),主循环每次喂狗。一旦固件因电机噪声耦合被卡死,看门狗超时复位 MCU → PWM 输出关闭 → **电机立即急停,不再狂奔**。彻底根治了"驱动途中固件挂死 → PWM 锁死 → 电机失控(runaway)"这一安全隐患。IWDG 复位后自动关闭,不影响 DFU 烧录。

### 3.4 故障取证(Fault Forensics)—— 挂死可诊断

- **复位原因识别**:启动读取 RCC 复位标志并清零,区分 IWDG(固件挂死)/ BOR(供电跌落)/ PIN(NRST 受扰)/ POR(上电)/ SW(软件)。
- **HardFault 捕获**:naked HardFault 处理程序捕获 CFSR/HFSR/BFAR/MMFAR + 堆栈 PC/LR,存入 `.noinit` RAM 段(暖复位不清),下次启动吐出。
- **明文取证行**:以 `[BOOT] CSR=... IWDG/BOR/PIN | HF=... CFSR/PC/LR` 明文经 CDC 每 5 s 持续发出,任何时刻连上都能读到上次复位真因。

此机制使"MCU 为什么挂/重启"从**靠猜**变为**可测**——挂死后重连即一眼定位是固件挂死、供电跌落还是崩溃,并精确到崩溃指令地址。

## 4. 底盘桥接与里程计(stm32_bridge)

### 4.1 桥接节点

`stm32_bridge` 订阅 `/cmd_vel`,以协议帧下发 STM32;读取 STATUS/ODOM 帧,发布 `/odom`、`/stm32/status`、`/joint_states`,并可发布 odom→base_link TF(导航时交由 EKF)。

### 4.2 里程计(麦轮正运动学)

由四轮编码器增量正解算机体速度 (vx, vy, wz),积分得位姿 (x, y, θ)。经现场标定:每转计数 1323、轮径 0.038 m,里程精度实测命令速度 ≈ 真实速度。

### 4.3 REP-103 标准坐标约定

里程计与命令通道严格遵循 REP-103 右手系(+x 前、+y 左、+wz 逆时针),bridge 作为 **REP-103 ↔ 固件约定的完整双向适配器**,保证 Nav2 与遥控发出的标准速度指令被正确执行、里程计以标准约定上报,为自主导航提供正确的坐标基准。

### 4.4 断链自愈(watchdog-reset resilience)

当 STM32 因看门狗复位而 USB 重枚举时,bridge **自动检测链路丢失 → 重开端口 → 重发 SET_MODE → 恢复通信**,并对里程计做重锚定(保留累积位姿、不产生跳变);同时对 STM32 上报的模式做自动补发,确保复位后底盘自动回到可驱动状态。**效果:偶发挂死从"整个导航栈崩溃"降级为"里程计打 2–3 s 嗝后自恢复"**,建图/导航可扛住偶发挂死。

## 5. 整车安全机制

| 机制 | 说明 |
|---|---|
| **独立看门狗** | 固件挂死 → 复位 → 电机急停(见 3.3),根治 runaway |
| **故障取证** | 复位原因/崩溃现场可读(见 3.4) |
| **断链自愈** | 链路复位后自动重连,栈不崩(见 4.4) |
| **雷达反应式避障** | 遥控/导航时按行进方向扇区最近障碍减速/停车(见 6、7) |
| **三重 deadman** | 前端松手发 0 → 后端时效判定 → 下位机命令超时,任一层断则停车 |
| **星形接地 + 去耦** | 功率地粗线直连电池负极、信号地单点汇合、TB6612 VM 就近大电解,抑制电机噪声耦合 |
| **命令超时急停** | STM32 端命令超时(无心跳/无指令)自动停车 |

**电气加固**:针对"电机大电流开关噪声耦合致固件挂死"的根因,完成整套加固——星形接地(功率地与信号地分离、仅在电池负极单点汇合)、STM32 5 V 轨大电解、两块 TB6612 的 12 V 输入各并大电解吸收开关尖峰、电机线远离信号线。配合看门狗与自愈机制,整车在电机负载下稳定运行。

## 6. 遥控功能(App 摇杆 + 雷达安全)

- **低延迟遥控通道**:App 摇杆(~10 Hz)→ 后端低延迟端点 → RDK `teleop_receiver` 拉取 → `/cmd_vel_teleop`。
- **雷达安全门控**:`lidar_safety` 按机体行进方向的扇区最近障碍**线性减速 / 近距停车**,旋转始终放行;后方近场自遮挡屏蔽(车尾结构不误判为障碍)。
- **平滑处理**:门控输出经中值滤波去抖,消除近场读数抖动导致的指令跳变,前进/后退均顺滑。
- **三重 deadman**:见第 5 节,网络延迟/断开不影响本地安全。

## 7. 建图功能(SLAM 一键在线建图)

- **一键启动**:`mapping.launch.py` 一条命令起雷达 + 里程计/IMU-EKF/TF + slam_toolbox + 遥控安全层。
- **在线建图**:slam_toolbox 异步模式,扫描匹配 + 回环检测,实时输出 `/map` 占据栅格与 map→odom 修正。
- **遥控建图**:App 摇杆遥控机器人巡场,地图从空白实时长出;支持随时存图。
- **数据可回放**:建图全程可录制 rosbag(/scan、/tf、/map、/odom 等),事后离线回放渲染"地图逐步建立过程"。
- **实测成果**:实验室实地建图成功,输出 `lab_map_v2` 地图(方向正确、闭环残差约 27 cm),备份至工作站。

**工程要点**:建图前停用其他重载服务腾出算力,保证 EKF/slam 不丢扫描;地图质量依赖里程计与扫描朝向的坐标一致性(见第 9 节技术攻关)。

## 8. 定位与自主导航(AMCL + Nav2)

### 8.1 一键自主导航

`nav.launch.py` 一条命令起:雷达 + 里程计/IMU-EKF/TF + **AMCL 定位** + **Nav2 全栈**(map_server / controller / planner / behavior / bt_navigator / velocity_smoother)。全部 lifecycle 节点自动激活。

### 8.2 AMCL 蒙特卡洛定位

基于存好的静态地图,AMCL 通过激光扫描匹配持续估计机器人位姿,发布 map→odom 修正。支持自动初始位姿或指定初始位姿。

### 8.3 Nav2 全局规划 + 局部避障

- **全局规划**:navfn 规划器在全局代价地图上生成到目标点的路径。
- **局部控制 + 避障**:控制器 + 局部代价地图实时融合激光,**动态避开路径上出现的行人/障碍**(移动家具无需入静态图,导航时实时避让)。
- **发目标**:`send_goal` CLI(nav2_simple_commander,setInitialPose + goToPose),打印实时反馈直至到达/失败/超时。

### 8.4 静态地图障碍处理(桌椅 keep-out)

针对实验室桌椅在激光高度仅有细腿、易与噪声混淆的问题:静态地图只建墙体与固定结构;移动桌椅交由 Nav2 局部代价地图实时避让;导航前将桌子整体占地在地图中标注为 keep-out 禁区(解决细腿钻缝与桌面在激光上方看不见两个问题),配合代价地图膨胀半径,保证规划安全。

### 8.5 自主导航实测

真机实测:一键起 Nav2 全栈(AMCL/控制器/规划器/bt_navigator 全部 active)→ 设初始位姿 → 发送目标点 → 机器人自主规划并前进到达,`result: SUCCEEDED`,里程计确认实际位移到位。**验证了完整链路:Nav2 规划 → 控制器发速度 → REP-103 经 bridge 翻译 → 固件 → 电机正确执行 → AMCL 定位 + 代价地图避障 → 自主到达目标。**

## 9. 关键技术攻关

本子系统在开发中攻克了一系列底层疑难问题,是系统稳定运行的基础:

1. **电机噪声耦合致固件挂死 / runaway**:定位真因为电机大电流开关噪声经共地耦合弹动 MCU;通过独立看门狗(急停兜底)+ 故障取证(可诊断)+ 星形接地与大电解(抑源)+ 断链自愈(软件容错)四管齐下根治。

2. **地图左右镜像**:定位真因为里程计与 IMU 偏航符号为左手系(反 REP-103);通过将麦轮正运动学的横移/偏航符号翻正、IMU 陀螺 z 取反,使全系统统一到 REP-103 标准,地图方向正确、定位闭环。

3. **定位漂移**:定位真因为实时可视化占用算力导致 slam 丢扫描;通过资源调度与离线回放渲染方案,兼顾地图质量与建图过程可视化。

4. **倒车卡顿**:定位真因为雷达安全层对后方自遮挡近场的门控逐帧抖动(非电机/机械问题);通过门控输出中值滤波 + 后方屏蔽半径整定解决。

5. **REP-103 命令约定统一**:使 Nav2 标准速度指令、遥控指令、里程计上报三者坐标约定一致,bridge 成为完整双向适配器,为自主导航奠定正确坐标基准。

6. **通信可靠性**:USB CDC 高速链路迁移 + 断链自愈,解决反复失联;里程计现场标定,保证命令速度与真实速度一致。

## 11. 结论

本机器人底盘运动与自主导航子系统已实现从**下位机实时电机闭环控制**到**上位机自主导航**的完整能力链,具备遥控、建图、定位、路径规划、动态避障、自主到达等核心功能,并配备独立看门狗、故障取证、雷达安全门控、断链自愈、星形接地等多重安全机制。系统采用标准 ROS 2 Humble + Nav2 架构,坐标约定严格遵循 REP-103,一键启动、模块化、可维护性强。真机实测:建图、定位、自主导航驱动全链路打通,可在真实实验室环境稳定运行,为上层巡检任务(感知检测、语音交互、云台指示等)提供了可靠的移动平台。
