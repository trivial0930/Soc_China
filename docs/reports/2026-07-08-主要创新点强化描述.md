## 创新点一:云—边—端"四层渐进式"分层决策架构(核心创新)

**痛点**:实验室巡检的危险判定,纯云端方案时延高、断网即瘫、隐私外流、按次计费;纯端侧方案算力不足,复杂场景误判多。现有巡检机器人多为"端侧检测 + 云端复核"两级,粒度粗:要么小事上云浪费带宽与费用,要么大事留在端侧误判。

**我们的做法**:构建 **L1 → L1.5 → L2 → L3 四层渐进决策**,按"事件复杂度"逐级升舱、逐级过滤:
- **L1(板端,毫秒级)**:YOLOv11s 经地平线 BPU(10 TOPS)NV12 硬件推理,10 类实验室危险物实时检出(3–4 FPS),与热成像融合出温度分级(见创新点二);
- **L1.5(板端,秒级)★ 本项目独有的"半层"**:在 RDK X5 的 8×A55 CPU 上以 llama.cpp 常驻一个 **2–3B 小型 VLM**(Qwen2.5-VL-3B / InternVL2.5-2B),对 L1 初筛事件做**低成本语义过滤**——常识性误报(常温插排、普通电线)直接在端侧闭环,不出板、不占网;
- **L2(边缘,~1.4s)**:局域网 Mac 上 Ollama 部署 **Qwen2.5-VL-7B**,对可疑事件做看图复核与动作决策(播报/复核/上报);选型经过实测淘汰:Qwen3-VL-8B 因强制思维链单次 ~35s 被弃用,体现"时延优先"的工程判断;
- **L3(云端,少量触发)**:阿里云百炼 **Qwen3-VL-Plus**,仅承担报告生成、多图综合等高价值低频任务。
层间由**升级门控(escalation gate)**控制:严重度、置信度、类别共同决定是否升舱,杜绝"全量上云"。

**与常规方案的本质区别**:
1. 不是"端+云"两级,而是**四层且含一个端侧小 VLM 中间层**——把"语义理解"这一原本必须上云的能力下沉到 10W 级嵌入式板卡,用 2–3B 模型过滤掉大部分本可不上云的事件;
2. 分层的依据不是简单阈值,而是**"检测置信度 × 温度证据 × 类别风险"的组合门控**,每层都产出结构化 JSON,可审计、可回放;
3. 全链路**断网可用**:L1+L1.5 在板端完全闭环,断网时告警本地暂存、恢复必达(见创新点四)。

**实证支撑**:L1 模型 P=0.8025 / R=0.7929 / mAP50=0.8178(独立混淆矩阵、PR/F1 曲线齐全);板端 BPU 实时 3–4 FPS;L2 温启动 ~1.4s;分层决策逻辑 200+ 单元测试覆盖;全栈(含 L1.5 llama-server)已纳入开机自启,真机重启验证通过。

**证据索引**:`evidence/01_危险对象检测/`(指标+混淆矩阵+真机图)、`docs/architecture/three_layer_hazard_decision.md`、`docs/architecture/l15_small_vlm_research.md`(选型研究)、`tests/test_cognition.py`/`test_tiered_cognition.py`/`test_escalation.py`。

## 创新点二:RGB—热成像跨模态"温度归属"危险判定

**痛点**:单目视觉只能看"是什么",看不出"烫不烫":一把断电的烙铁和通电 300℃ 的烙铁在 RGB 里几乎一样;纯热成像只能看"哪里热",不知道"热的是什么"——反光、阳光直射、暖气都会误报。

**我们的做法**:80×62 辐射测温热像仪(SenXor MI48)与 RGB 相机做**像素级空间配准**(热点质心 ↔ YOLO 框中心对应点对,RANSAC 求解仿射/单应矩阵,中心区重投影误差 ~10px,写入 `thermal_rgb_calib.yaml`),在此之上实现**"温度归属"判定**:
- 每个 YOLO 检出框被赋予实测温度 → 输出「**类别 × 温度状态**」的组合语义:`soldering_iron (active 280C)` 是 CRITICAL,`soldering_iron (cold 32C)` 只是 WARNING;
- **无主热点(orphan hotspot)单独告警**:热像里发现高温区但 RGB 框不住任何物体 → 报"unidentified heat source + 温度",覆盖"看不见的危险"(遮挡下的发热、阴燃);
- 工程细节:死像素(raw-0 → −273℃)中位数替换 + 3×3 中值滤波、2/98 百分位归一化、两相机倒装的方向矫正,保证真机长稳。

**与常规方案的本质区别**:常见做法是"热像超阈值就报警"或"两路画面并排显示靠人看"。我们输出的是**物体级、带语义的温度归属结论**——同一个物体因温度不同得到不同风险等级,同一个温度因归属物体不同得到不同处置,并把"有物体的热"与"无主的热"分成两类告警。这直接决定了后续分层决策的输入质量。

**实证支撑**:真机标定验证(power_adapter "active 64C" 正确归属到检测框);实验室实测输出如 `WARNING: soldering_iron (cold 32C)`、`CRITICAL: hot_air_gun (active 120C)`;板上累计 90 条融合事件 JSON(类别+置信度+温度+严重度)+ 90 张证据图;融合管线 5 个专项测试文件覆盖。

**证据索引**:`evidence/02_热像测温与融合/thermal_rgb_calib.yaml`、`docs/validation/daily/2026-06-07-thermal-90-bringup.md`、`rdk_x5/ros2_ws/src/thermal_detector/`(fusion.py/hazard_pipeline.py + tests)、板上 `evidence/`+`events/`(90 组)。

## 创新点三:全离线端侧语音交互 + "三源同路"的统一命令执行

**痛点**:巡检现场常无公网,云语音方案(小度/天猫精灵式)不可用;而机器人往往"能听的"和"能做的"两套系统割裂——语音一套逻辑、App 一套逻辑、自动决策又一套,行为不一致、维护成本三倍。

**我们的做法**:
1. **全离线语音链路**:板载麦克风 → sherpa-onnx **KWS 唤醒**(「小巡」/「巡检助手」)→ **silero VAD 断句** → **SenseVoice 中文 ASR** → 意图理解 → 执行 → **Matcha TTS 中文回执**(RTF~0.4,约 2s)。全链在 RDK X5 上运行,零云依赖;
2. **"规则优先 + 小模型兜底"的双层意图理解**:高频指令(复核/激光/巡检/验收/播报)走毫秒级规则匹配(含中文数字工位号归一);规则不中再降级给 L1.5 小 VLM 解析成命令 JSON(带置信度门限)——**复用创新点一的端侧模型,一模两用**;90%+ 指令不触发大模型,算力占用可控;
3. **三源同路(核心工程创新)**:语音指令、手机 App 下行指令、自动决策动作,统一收敛到同一个 `dispatch_command()` + 共享 `CommandExecutor` 执行——**三个入口、一套语义、一条执行路径**,任何能力(激光指示、播报、验收、复核)三个入口天然同权、行为一致;
4. **诚实降级**:依赖底盘移动的指令在移动未打通时,回执如实说"已记下,底盘调试中稍后执行",不假装完成——人机信任设计。

**与常规方案的本质区别**:不是"接一个语音 SDK",而是把语音作为**与 App、自动决策平权的第三个命令源**接入统一执行总线;意图理解不是纯 NLU 模型也不是纯关键词,而是分层混合;且全部离线、可开机自启、App 可远程一键关闭监听(释放算力)。

**实证支撑**:59 个语音相关单元测试全绿(状态机/意图/话术/执行);上板真机 9 节点开机自启重启验证通过;真人唤醒-指令-回执链路实测;TTS 时延 ~2s 实测。

**证据索引**:`docs/superpowers/specs/2026-06-21-voice-asr-interaction-design.md`、`docs/architecture/voice_asr_setup.md`、`rdk_x5/ros2_ws/src/inspection_manager/`(intent/dialog/asr_controller/command_executor + tests)、`docs/reports/2026-07-06-语音交互与App功能开发及实验室实测.md`。

## 创新点四:面向"真会出事"的多层纵深安全体系

**痛点**:移动机器人事故几乎都发生在异常路径:遥控断线、程序挂死、固件跑飞、断网丢告警。多数学生项目只做"正常路径",安全是贴纸不是设计。

**我们的做法**——从手机到电机、从应用层到固件层,五道独立防线:
1. **App 端 deadman(三重)**:摇杆松手 / 离开页面 / App 切后台 / 触摸中断,任一发生**立即连发零速度**(多发抗丢包);速度硬夹在底盘限速内。19 个单测锁定该行为;
2. **机器人本地雷达门控**:遥控速度不直通电机,先经 `lidar_safety` 反应式门控——前方障碍分级减速直至停止、旋转放行、后方近场自遮挡屏蔽;网络时延不影响避障实时性(避障在板端本地闭环);
3. **STM32 独立看门狗(IWDG,~250ms)**:固件跑飞 → 硬件复位并关断 PWM 急停,从根上封死"电机 runaway";配套 HardFault 取证(启动时打印崩溃现场寄存器),挂死可诊断;
4. **USB 链路断链自愈**:RDK↔STM32 桥接节点跨重枚举自动重连,链路故障从"整栈崩溃"降级为"里程计 2–3 秒打嗝后自恢复";
5. **断网告警必达(store-and-forward)**:告警/简报持久化重传队列,断网期间零丢失、恢复后全量补传,后端按 event_id 幂等去重不重复;上板实测断网 45s 零丢失。

**与常规方案的本质区别**:五道防线**分属不同层级、互相独立**——App 层失效有雷达层,软件层失效有固件看门狗,任何单点故障都不会同时击穿两层。"安全"不是某个功能,而是贯穿架构的纵深设计。

**实证支撑**:teleop 门控 11 测 + App deadman 19 测 + uplink 补传 15 测;IWDG/断链自愈有 git 提交与真机验证记录;断网 45s 补传实测零丢失。

**证据索引**:`rdk_x5/ros2_ws/src/teleop_safety/`、`tests/test_teleop_gate.py`/`test_uplink.py`、`docs/validation/daily/2026-06-27-pcb-bringup-teleop-lidar-safety.md`、`docs/reports/2026-07-07-底盘运动与自主导航功能报告.md`(IWDG/取证/自愈)。

## 创新点五:无公网依赖的全场景组网与管理闭环

**痛点**:实验室/竞赛现场网络不可控:没有固定 Wi-Fi、手机热点频段受限、Mac/机器人/手机三端不一定同网段。多数系统默认"都在同一局域网",一换场地就瘫。

**我们的做法**:
1. **三级组网策略**:同局域网直连 → **Tailscale 虚拟组网**(三端各自能上网即可互通,手机离网实测可用)→ 随身路由中继(GL.iNet 挂手机热点,机器人 eth0 上网,后端可整体搬到板上本地跑,App 经路由端口转发访问,完全不出公网);
2. **管理端 App 全闭环**:秒级 SSE 实时告警(critical 弹通知)+ 证据图、工位/验收/报告、**7 类远程命令下行**(播报/激光/报告/复核/巡检/验收/寻物)、**虚拟摇杆遥控**(10Hz+避障状态条)、**一键建图模式**(App 开关→机器人自动腾资源起建图栈→摇杆绕场→一键存图,开关状态以机器人真实回报为准)、语音开关、播报音量;
3. **远程运维实证**:机器人带到实验室后,全部排查(接线故障定位、热成像修复、识别验证、服务起停)均经 Tailscale **远程完成**,现场只动了螺丝刀。

**与常规方案的本质区别**:App 不只是"显示大屏",而是覆盖"看(告警/证据)—操(遥控/命令)—建(建图存图)—管(开关/音量)"的**完整操作闭环**;组网不押注任何单一网络条件,三级方案覆盖从理想到最恶劣的现场。

**实证支撑**:手机离网经 Tailscale 使用 App 实测通过;实验室实地"App 遥控→SLAM→存图"全链路真机跑通(地图 ~20.6×26.4m);实验室远程排障全程记录;App debug+release 出包(release 53.9MB),analyze 零问题。

**证据索引**:`docs/reports/2026-07-06-语音交互与App功能开发及实验室实测.md`、`evidence/03_自主导航/`(地图+建图视频)、`app/API_SPEC.md`、`app/mobile/`。

## 创新点六(支撑性):可验证的工程化方法

> 建议作为"工程质量保障"陈述,支撑上述创新点的可信度,不单独作为学术创新点。

- **纯逻辑注入式架构**:意图/话术/状态机/命令执行/融合/门控等核心逻辑全部与 ROS、推理库、硬件解耦,依赖注入,**可在无硬件的开发机上单元测试**——全仓库累计 300+ 单元测试(分层决策 200+、语音 59、遥控 19、门控 11、补传 15 等);
- **规范研发流程**:关键功能按"设计文档 → 实现计划 → 逐任务 TDD → 双阶段代码复审 → 全分支复审"推进,全分支复审曾捕获逐任务测试漏掉的真机级集成缺陷(语音节点漏注册命令发布器);
- **故障可诊断设计**:固件 HardFault 取证、事件全链 JSON 可回放、建图过程 rosbag 留存、远程判据文档化(如"温度全 −273=SPI 断 / I2C 不 ACK=供电断")。

## 附 A:一句话版(报告摘要/PPT 标题可直接用)

1. **分层决策**:提出"云—边—端四层渐进决策"架构,在 10W 级嵌入式板卡内嵌 2–3B 视觉语言模型作为端侧语义过滤层,实现"毫秒检测、秒级理解、按需上云",断网可全功能本地闭环。
2. **多模态融合**:实现 RGB—热成像像素级配准与**物体级温度归属**判定,输出"类别×温度状态"组合语义(同物不同温、同温不同物 → 不同风险),并独立识别"无主热源"。
3. **语音交互**:全离线中文语音链路(唤醒—断句—识别—意图—合成)上板运行,"规则优先+端侧小模型兜底"双层意图理解;语音、App、自动决策**三个命令源统一到同一执行总线**。
4. **纵深安全**:App deadman、雷达本地门控、固件独立看门狗、链路断链自愈、断网告警必达——五道跨层级独立防线,任何单点故障不击穿系统。
5. **组网与闭环**:局域网/Tailscale 虚拟组网/随身路由三级组网,任意现场可用;手机 App 覆盖"看—操—建—管"完整管理闭环,支持远程遥控与一键建图。

## 附 B:强化表述的使用建议

- **主打顺序**:答辩/报告建议以 **创新点一(分层决策)** 为第一主打——它是架构级创新且与"云边端协同"命题最契合;**创新点二(温度归属)** 为第二主打——最直观、有真机演示;三、四、五按篇幅取舍。
- **每个创新点的标准讲法**:一句痛点 + 一张架构图/实拍图 + 一个量化数字(mAP 0.82 / L2 1.4s / 断网 45s 零丢失 / TTS 2s / 59+19+11 测试)+ 一句"与常规方案的区别"。
- **避免的写法**:不要写成功能罗列("我们做了 A、B、C")——创新点强在"**为什么这样做、别人为什么不是这样**";已明确未实现的(固定端—移动端协同、批量导航统计实验)不要写入创新点,防答辩被问穿。
