## 四、上板抓到并修复的真 Bug(事项 5 核心 —— 单测抓不到,只有真机暴露)

| # | 现象 | 根因 | 修复 |
|---|---|---|---|
| 1 | `command_receiver_node` 一起就崩 | `self.executor=CommandExecutor(...)` 撞 `rclpy.Node.executor` 保留 property(setter 调 add_node) | 改 `self._cmd_executor` |
| 2 | KWS 构造报错/唤醒空 | sherpa 1.13.3 无 `KeywordSpotterConfig`;模型文件名带 epoch;`get_result` 需在解码**循环内**逐帧查 | 扁平构造器 + `_resolve_onnx` + 循环内 get_result + reset_stream |
| 3 | 采集全静音 | 麦是**独立 USB 麦 hw:2,0(仅48kHz)**,非板载 ES8326;ES8326 采集 ALC 曾关 | `mic_device:"USB Composite Device"` |
| 4 | 命令识别恒为空 | `vad.front` 是**属性非方法**(`front()`→TypeError);`segment.samples` 是 **list**,直接喂识别器返回 `''` | `front`(无括号)+ `np.asarray(float32)` |
| 5 | 段间歇 `rms=inf`→命令丢 | 48k→16k 用 **8 阶 IIR**,一次毛刺的 inf/nan 永久污染 IIR 无限记忆 | 换 **FIR(firwin+lfilter)** + `nan_to_num` 兜底 |
| 6 | **每句延迟 17 秒** | `sherpa-onnx-offline-tts` CLI **每次重载模型** | **常驻 TTS 服务** `/root/tts_server.py`(加载一次→0.5s/句),`sherpa_say.sh` 改写 `/tmp/tts.fifo` |
| 7 | 越说越乱、自我循环 | 音响播报被麦拾到→自我识别 | **半双工 anti-echo**:daemon 播放时 touch `/tmp/tts_playing`,poll 期间静音麦 +0.4s |
| 8 | USB 麦反复掉线致崩 | USB `autosuspend=2s` 太激进 + 设备不稳 | `echo -1 > /sys/module/usbcore/parameters/autosuspend` + 开麦失败重试 60s 不崩 |
| 9 | 唤醒时灵时不灵 | 推理占 GIL→音频回调饿死→PortAudio xrun 丢帧 | InputStream `latency="high"`(大缓冲吸 GIL 停顿,xruns=0) |

**调优**:`kws_threshold` 0.25→0.10;命令须带触发词("**播报**请大家注意安全");连续重采样(非逐块 resample_poly,后者太重致积压)。最终用户实测**"现在很顺"**。

### 6.2 `set_volume`(事项 13)—— App 调 TTS 播报音量
USB 音响**无 ALSA playback 音量控件**(只 Mic/AGC),走**软件增益**:`POST /api/commands {set_volume, level:0-100}`→后端校验→command_receiver 写 `/root/.tts_volume`→TTS 服务读它 `ffmpeg volume=4.5*level/100`(0 静音、100 最响)。CommandExecutor 加可注入 `set_volume` 回调。**室友睡觉时先只搭建+非音频链路验证**(命令→文件写入→增益计算),醒后**真实三档播放实测响度递增,通过**。API_SPEC v1.3。App 滑块由另一 agent 做。

### 6.3 teleop 低延迟遥控通道(事项 16)
`app/backend/teleop.py`(纯 `TeleopStore`,6 单测)+ server 4 端点:`POST/GET /api/robot/teleop`(速度,Bearer,后端 clamp vx,vy∈±0.4/wz∈±1.5,`age_ms` 供 deadman)、`POST/GET /api/robot/teleop/status`(雷达状态)。与 2s 命令队列完全分开、内存覆盖式。契约自测全过(clamp、age_ms、401、往返、命令队列 200 不受影响)。API_SPEC §4.7 + v1.4。

### 6.4 建图模式(事项 17)
`commands.py` 白名单 `set_mode {mode}`/`save_map {name?}`(**不做业务校验**,合法性 RDK 判)。`teleop.LatestValue`(通用「最新值+age_ms」)复用做 mode store;server `POST/GET /api/robot/mode`(RDK 每 tick 上报真实 mode,App 读 `{mode,age_ms}` 判在线)。事实源是 RDK。自测:set_mode 入队被 live RDK 消费(e2e)、save_map 401/201、mode 心跳 age_ms 增长。API_SPEC §4.8 + v1.5。**注**:当前板上 command_receiver 尚无 set_mode 执行逻辑(在 `feat/chassis-bringup-stack` 未 rsync 上板),故测试命令被 ack 但未真切换,机器人安全留在语音模式;后端契约本身正确。

## 八、答疑 / 分析类事项(事项 10-12, 14, 19)

- **温度(事项 10)**:`awk '{printf "%.1f°C\n",$1/1000}' /sys/class/thermal/thermal_zone0/temp`;列全温区遍历 `thermal_zone*`;实跑 CPU ~38°C。阈值参考 75°C。
- **L1.5 是否纯文本(事项 11)**:读代码确认 **是**——`cognition.yaml: fast(L1.5) send_image:false`,`cognition.py` 据此不带图、prompt 纯文本;因 A55 看图 ~70s 太慢,靠 L1 结构化事实判,图像验证交 L2。模型本身(qwen2.5vl:3b)有视觉能力,是**配置**关掉图;`send_image:true` 可开(代价 ~70s/次)。
- **L1.5 内存/mmproj(事项 12)**:4G ≈ LLM权重~1.8G + mmproj视觉投影~1.3-2.0G + KV~0.3-0.7G。**纯文本时 mmproj 纯浪费**,去 `--mmproj` 省 ~1.5G、零功能损失、可逆(加回参数即恢复)。解释 mmproj=把图像编码成 LLM 能读的 token 的翻译器;"关掉(去启动参数)vs 删文件"区别。**该运行时 `start_llm.sh` 归另一 agent,我只给建议未动**。
- **App 连不上(事项 14)**:后端本机 200、防火墙关、监听 0.0.0.0。给地址:同 WiFi `http://192.168.5.53:8000`(LAN IP DHCP 会变)/ Tailscale `http://100.114.38.16:8000`;写 token `~/.app_ingest_token`。定位主因=**Tailscale 两端离线**(手机上次在线 1 天前)。
- **L1/L1.5/L2/L3 部署非本轮我做(事项 19)**:认知框架代码(4 层,241 测试)在更早会话建;L1.5 板上运行时(llama-server:8080、start_llm.sh、cognition_node 自启)是**另一 agent** 做的;我本次只读代码答疑 + 给 mmproj 建议,未部署。
