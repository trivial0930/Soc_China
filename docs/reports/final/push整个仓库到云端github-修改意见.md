## 二、与代码现状不符/内部矛盾的事实表述

### 2.1 IMU(BMI088)表述自相矛盾,且落后于实际进度——这是把已验证成果写丢了

报告三处说法互相打架:

| 位置 | 说法 |
|---|---|
| 2.2.1 | "实物接入和安装外参待确认……不把IMU作为已完成导航证据" |
| 表 2 | "IMU接口预留" |
| 表 3 | "BMI088 …… IMU运动姿态控制"(写得像已完成) |

**实际状态(有上板验证记录):** BMI088 陀螺已实物接入 I²C,`mapping.launch.py` 中轮式里程计 + IMU 陀螺 yaw 率经 EKF 融合,**实地建图(lab_map / lab_map_v2)就是带 IMU 融合跑出来的**;accel 曾因模块焊点缺陷间歇掉线,已补焊修复并完整断电 POR 后验证全功能 ok;IMU 节点带 gyro-only 降级(accel 再掉也不崩、REP-145 标缺失,host 4 测 + 上板验证)。

**建议:统一改为——"BMI088 陀螺 yaw 率已接入 EKF 并用于实地建图;加速度计具备降级容错(掉线自动 gyro-only,不阻塞建图)"。** 这从"预留"变成实测成果 + 容错设计双加分项,现在的写法是自贬。同时修掉表 2/表 3 的矛盾。

### 2.2 断网补传的表述可以更准确

2.3.4 说"该队列尚未落盘……本文表述为短时断网恢复补传"。"尚未落盘"属实,保留是对的;但现状是 RetryQueue 对 event/brief 设了 persistent_kinds,**进程存活期间无限重试、恢复后全量补传**(45 s 故障注入 0 dropped,表 6 已列),普通记录才是 5 次约 25 s 放弃。建议改为:"关键告警与简报在进程存活期间不设重试上限,断网期间不丢、恢复即补传;队列尚未落盘,进程重启不保证恢复"。比"短时"更准确也更有含金量。

### 2.3 TTS 数字前后不一致

2.3.6 写"常驻后单句响应由约 17 s 降至约 0.5 s",表 1/表 6 写"RTF≈0.4,示例语句约 2 s"。两处口径不同(0.5 s 疑为短指令回执、2 s 为示例长句)。建议统一为一个口径,或写明"短指令回执约 0.5 s,长句示例约 2 s(RTF≈0.4)"。

## 三、建议补入的新成果(均已实现并在本次推送的提交里)

### 3.1 2.3.5 安全链路缺了最有力的三件套

表 4 / 2.3.5 只写了超时停车与 CDC 重连。以下均已实现、有提交号、且直接支撑"可恢复安全链路"的核心主张,建议补进正文或表 4:

- **STM32 独立看门狗 IWDG**(ddae20c4):固件挂死约 250 ms 内硬件强制停车——这是"先保证停车"的最后防线,比软件超时更硬;
- **复位原因 + HardFault 取证经 CDC 上报**(578c306c、7c557b6f):每次异常复位可事后定位原因,支撑"可验证"主张;
- **桥接跨 USB 重枚举自动重连 + 里程计重锚定 + SET_MODE 自动重发**(13203488、8c697ff1):对应表 6"通信恢复约 2~3 s"那行的实现机制,目前正文只写了结果没写机制。

### 3.2 2.3.5 可补底盘 runaway 的工程证据

4.2 心得已提过一句"零指令空转来自编码器方向错误"。建议在 2.3.5 或 3.3.3 用一句话给出闭环:根因为搬运后 LF/LR 编码器接反致 PID 误补速,修复 = 交换定时器映射 + 0.10 rad/s 零指令死区守卫,架空四轮验证(前进全正 / 静止全零)。这正呼应报告"每项功能可从结论追溯证据"的方法论。

### 3.3 2.3.6 App 建图模式可写实机制(本对话刚提交的代码)

现在只有"建图模式、地图保存"六个字。已实现且可写的:模式切换以 `/root/.robot_mode` 为板上事实源,后端 `GET /api/robot/mode` 心跳(路由已挂,server.py:252/258 核实),**App 以 2 s 轮询机器人回报为准、开关不自欺,>10 s 不一致自动重发一次**;切换脚本一键停语音栈腾 CPU 再起建图栈,命令通道常驻不杀。这套"以机器人状态为事实源"的设计和 2.3.6 现有表述完全一致,值得展开两三句。

### 3.4 表 5 teleop_safety 可补自遮挡屏蔽

现只写"减速/停车后的安全速度"。实际还有:机身后方近场自遮挡屏蔽(near_masks,默认后向 45° / 0.30 m)解决雷达照到自身结构误触发、倒车判抖修复。一句话即可,是真实工程细节。
