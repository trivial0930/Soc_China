## 一、现状诊断:为什么现在的创新点"不够强"

报告 1.5 现有三条:
1. 自主导航与云台多模态感知协同
2. 云—边—端 AI 按复杂度分层
3. 感知、处置与复检一体化

问题不在方向,而在**表述**——三条都是"做了什么"的名词罗列,缺四样评委真正在意的东西:

| 缺什么 | 后果 |
|---|---|
| **真问题** | 看不出为什么值得做 |
| **与现有/常规做法的差异** | 像"把成熟模块拼起来",显不出创新 |
| **可复述的机制** | 只有结论没有"怎么做到的" |
| **量化/可追溯证据** | 无法与"概念 PPT"区分 |

**强化原则**:每条创新点写成一个"问题—反例—机制—证据"的小论证,并明确指出**系统级**的贡献(不是"用了 YOLO / Nav2 / VLM",这些谁都能用;而是把它们**在算力、带宽、隐私、安全约束下组织成可恢复、可降级、可追溯的闭环**)。

## 二、建议的四条(可直接替换 1.5)

> 建议从 3 条扩为 4 条,并把"安全/容错运动"从散落的工程细节**提炼成一条独立创新点**——这是本作品区别于"演示级 demo"的核心,却被现稿严重低估。

### 创新点 1|复杂度驱动的"云—边—端"认知分层与逐级降级

**问题**:实验室巡检要在**嵌入式算力、局域网带宽、数据隐私**三重约束下做出可靠判断——把所有画面传云端既不实时也不合规,全塞进端侧又跑不动大模型。

**常规做法的局限**:多数方案要么"端侧只做检测、云端做一切"(依赖网络、隐私差、离线即瘫),要么"端侧硬跑大模型"(帧率/时延不可接受)。二者都是**单点式**,任一环节(网络/模型/算力)不可用即失效。

**本作品的新机制**:按**事件复杂度**四级路由 —— L1 端侧 BPU(YOLO11s + 热像,持续实时初筛)→ L1.5 端侧轻量 VLM(常见异常本地语义复判)→ L2 局域网工作站多模态理解(Qwen2.5‑VL)→ L3 云端按需归纳(Qwen3‑VL‑Plus)。**关键在于"只上传最小必要证据 + 逐级降级"**:正常数据本地过滤、普通异常本地闭环处置,只有复杂/周期性事件才上传经裁剪脱敏的最小证据包;任一上层不可用时**自动回退到下一层直至规则模板**,本地告警与处置永不阻塞。

**证据**:BPU 检测约 3~4 FPS;本地边侧 Qwen2.5‑VL 7B 热调用约 1.4 s;云/边不可用时后端回退规则模板、不阻塞本地链路(已具备 mock/local_vlm/tiered 三配置)。

**一句话卖点**:不是"多模型堆叠",而是**"算力—带宽—隐私三约束下的成本自适应认知,断网断模型都不瘫"**。

---

### 创新点 2|"AI 只建议、绝不驱动"的硬安全边界 + 多层容错运动架构 ★(建议重点强化/新增)

**问题**:移动机器人一旦让"上层智能"直接控制底盘,任何模型误判、网络抖动或软件卡死都可能变成**物理失控**;实验室场景对安全的要求高于对智能的要求。

**常规做法的局限**:很多作品把导航/避障/决策都放在 Linux 应用层甚至云端,安全依赖"软件正常运行"这一脆弱前提——进程卡死或网络断开时,电机可能保持最后指令继续运动(runaway)。

**本作品的新机制**:把安全做成**独立于 AI 与网络的多层硬边界**:
- **控制下沉**:确定性四轮 50 Hz 闭环下沉到 STM32 MCU,不依赖 Linux 应用进程;
- **三级超时停车**:App 松手/离页 → RDK 桥接 0.5 s 清零 → STM32 命令 0.5 s / 心跳 2 s 停车;
- **独立看门狗(runaway 根治)**:STM32 IWDG(~250 ms),固件被电机噪声冻住停止喂狗 → MCU 复位 → PWM 关闭 → **电机急停**——比超时停车更强,因为它在**固件本身卡死**时仍能停车;
- **故障取证(可诊断)**:复位后经 CDC 输出 `[BOOT] CSR=… IWDG/BOR/PIN/POR/SW | HardFault CFSR/PC/LR`,把"为什么挂"从猜变成可测;
- **断链自愈(可恢复)**:USB CDC 重枚举后桥接器自动重连、重发模式、里程计重锚定(保留位姿不跳变),偶发挂死从"整栈崩溃"降级为"里程计 2~3 s 自恢复";
- **AI 权限边界**:大模型只返回**结构化建议**,事件编号/工位/时间/隐私字段校验不过立即回退本地模板;**大模型不能直接控制底盘或关闭事件**,底盘速度始终由 Nav2 + 雷达安全门控 + STM32 三方共同约束。

**证据**:强制 USB unbind/rebind 两次,日志"link lost→reconnect→里程计续上"、节点全程不崩;看门狗架空/实测挂死→复位→停机(无 runaway);取证行实测可读并正确区分复位原因;命令与里程计两侧均已 REP‑103 双向适配、实测方向一致。

**一句话卖点**:**"智能可选、安全必达"——AI 与网络负责变聪明,一套下沉到 MCU 的独立看门狗/取证/自愈/超时机制负责绝不失控。**

> 这条是把本作品与"跑得起来的 demo"拉开差距的关键:多数参赛作品能演示导航,但很少能证明"固件卡死时电机会安全停下且可回溯原因"。强烈建议作为核心创新点之一充分展开。

### 创新点 3|RGB—热像对象级融合的风险分级(而非单像素测温)

**问题**:热成像单看最高温像素极易误报(反光、人体、阳光都可能是热点),纯视觉又判断不了设备是否**真的**在发热。

**常规做法的局限**:直接用"最高温 > 阈值"报警,或仅凭视觉外观判断,误报/漏报都高。

**本作品的新机制**:RGB 目标框与 Thermal‑90 温度矩阵经**镜像/180°旋转/3×3 单应性标定**对齐到同一对象坐标,**只有"视觉对象 + 热点位置 + 温度 + 持续性"共同满足规则**才升级风险;无法关联但显著发热的区域标为"未识别热源"交人工复核——既不被单像素带偏,也不被外观误导。

**证据**:词表外仪器场景可输出 objects=0(不乱报);通电发热对象形成"目标框—热点"关联并输出风险等级与实测温度;仅热点无对象时输出 unidentified heat source。

**一句话卖点**:**"对象级证据融合 + 持续性门控",把热成像从'像素报警'升级为'可解释的风险判定'。**

---

### 创新点 4|以 event_id 贯穿的可追溯闭环("先定义证据,再实现功能")

**问题**:巡检类系统常见"功能孤岛"——检测、告警、处置、验收各存各的,事后无法复盘"这条告警到底怎么来的、处理没有"。

**常规做法的局限**:结论只停在界面,追不回原始证据,难以审计,也难以复现。

**本作品的新机制**:把"感知—处置—复检"组织成**同一 event_id 贯穿**的可追溯事件链——任务派发、导航状态、RGB/热像证据、人工整改、复检关闭全部挂在同一事件上;每项结论都能从界面追溯到**同一版本**的配置、日志、图片或地图文件;断网时关键告警进重试队列,恢复后按 event_id 幂等补传不重复。

**证据**:事件服务(FastAPI/SQLite/SSE)按 event_id 幂等写入;断网 45 s 恢复 0 丢失补传;地图/模型/配置/日志均可版本化回查。

**一句话卖点**:**"把可追溯性做成架构约束",让每个结论都能回到证据,而不是停在界面。**
